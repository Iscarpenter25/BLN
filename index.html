What We Need & Why

Today, data flows between Bell and Silver Fern rely on manual file exchanges — Excel workbooks and CSVs. Moving this data into structured Business Central tables enables automated, incremental synchronization — eliminating manual handoffs and ensuring both systems stay current and accurate with a single system of record.

Silver Fern Planning Systems

Two Data Flows We Need From Business Central

Store → Facility Mapping — Which distribution facility serves each HD store location, by product program. Used for demand allocation and distribution routing.

Variant Mix Allocation — Weekly mix percentages for each production variant, by facility. Used for recipe construction and production scheduling.

Key Takeaway

Two New BC Tables

We are proposing two new tables in Business Central: Store-to-Facility Mapping and Variant Mix Allocation. The Variant Mix table does double duty — it captures both the standard weekly percentages and regional mix differences (when one facility has different percentages than others) in a single table. No additional table is needed for regional exceptions.

BC Table 1

Store-to-Facility Mapping

Maps each HD store location to its default distribution facility for a given product subclass. This tells our planning systems where demand for each store should be routed.

What This Replaces

Today: Manual

Today, we maintain this data through custom site mapping spreadsheets. When programs are added or facility assignments change, we receive an updated file from Bell, manually parse it, and load it into our system each time.

With this data in a BC table, changes flow automatically through the standard data sync — no file handoffs, no delays, and one system of record. If someone needs to change a store's facility assignment, they can go directly into BC and update it without having to come to us first.

Proposed Table Structure

Business Central

We discussed a table with the following key columns:

  • StoreCode — The HD store code (e.g., HD0105, HD2003)
  • SubclassCode — Product subclass or program (e.g., ANNUALS, MUMS, TROPICALS)
  • DefaultFacilityCode — The distribution facility that serves this store for this subclass (e.g., KY01, VA05, VA07, NC04, OH08, VA06, NRS)
  • LastModifiedDate — Timestamp so we can pull only records that have changed since our last sync

BC Table 2

Variant Mix Allocation

Weekly mix percentages for each production variant, broken out by facility. This drives recipe construction and production scheduling in our planning systems.

What This Replaces

Today: Manual

Today, Bell maintains variant mix allocations in the "4M — Variant Mixes" Excel workbook. The file has separate sheets per facility (e.g., KY01, EVERYONE ELSE) with 52 weekly columns showing the percentage allocation for each variant.

When allocations change, the updated workbook is exported and sent to Silver Fern. We parse it, transform the weekly columns into date-ranged records, and load it into Produce. This is a full replacement each time — there is no incremental update.

What We Receive Today (Excel Format)

The Excel has fixed identifier columns followed by 52 weekly percentage columns:

Location Common Item Prod. Item Variant WK 1 WK 2 ... WK 48 WK 49 WK 50 WK 51 WK 52
KY01 1000190 4000899 4000899B01 20% 20% ... 20% 40% 40% 40% 20%
KY01 1000190 4000899 4000899B02 20% 20% ... 20% 20% 20% 20% 20%

Highlighted cells show a seasonal shift — variant B01 increases from 20% to 40% during weeks 49–51.

Proposed Table Structure

Business Central

Instead of 52 weekly columns, each week becomes its own row — a standard normalized format that is easy to query, filter, and sync incrementally.

Column Type Description
EntryNo PK Integer Auto-increment primary key
LocationCode FK Code[10] Warehouse / facility (KY01, VA06, etc.)
CommonItemNo Code[20] Sales item number (1MM — e.g., 1000190)
ProductionItemNo FK Code[20] Production item number (4MM — e.g., 4000899)
VariantCode Code[20] Production variant (e.g., 4000899B01, or * for wildcard)
WeekNo Integer Calendar week number (1–53)
MixPct Decimal Allocation percentage (0–100)
SeasonYear Integer Planning year (2026, 2027)
LastModifiedDate DateTime Timestamp for incremental sync

Example Records

Location Common Item Prod. Item Variant Week Mix % Year
KY01 1000190 4000899 4000899B01 1 20 2026
KY01 1000190 4000899 4000899B01 49 40 2026
KY01 1000190 4000899 4000899B02 1 20 2026
VA06 1000190 4000899 4000899B01 1 25 2026
Notice the LocationCode column. The same item can have different mix percentages at different facilities. In the example above, variant B01 is 20% at KY01 but 25% at VA06 for the same week. This is how regional mix differences are captured — no separate table required.

Design Notes

Why one row per week? The current pivoted Excel format (one column per week) is difficult to query and sync. One row per location/item/variant/week is a standard normalized structure — easy to filter, join, and pull via API.

Why SeasonYear? This allows Bell to prepare next year's mix plan without overwriting the current one. Our system can pull the active season while Bell preps the next.

Why LastModifiedDate? This enables incremental sync. Instead of replacing the full dataset every time, we query for records modified since the last sync and pull only the changes.

Wildcard handling: Items that don't split by variant use * in the VariantCode column with 100% allocation. Same table, same structure — no special case.

Production Item Regions — Resolved by Table 2

Regional Mix Differences

This is what we referred to as "ProductionItemRegions" during our call — the piece that was difficult to explain without the visual context. Here is a clearer breakdown of what it is and how it gets resolved.

On our side, we maintain a mapping that tracks when a production item has different mix percentages depending on the region. For most items, every facility gets the same mix. But for a small number of items, the weekly percentages are different between facilities — for example, KY01 (which serves Central and Midwest stores) may get a cold-hardy variety at 100%, while other facilities get a blended mix.

Today, this is embedded inside the variant mix Excel as separate sheets per facility (e.g., "KY01" and "EVERYONE ELSE"). There is no standalone file for these regional exceptions — it is implicit in the fact that the two sheets have different values for the same item.

This is resolved by the Variant Mix Allocation table proposed above. Because that table includes a LocationCode column, facility-specific differences naturally show up as separate rows. Our system reads those differences and handles the regional routing internally. No additional BC table is needed.

Bell's team continues to work the same way they already do — maintaining different mix percentages per facility — just in a BC table instead of separate Excel sheets.
