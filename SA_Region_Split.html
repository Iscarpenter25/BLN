South Atlantic Region Split

BLN Forecast — Testing Plan & Migration Strategy
Internal • Updated March 6, 2026 • QADB_FORECAST (testing) / BLN_FORECAST (production)

Context

Bell Nursery plans to split the SOUTH ATLANTIC region into 4 sub-regions: South Atlantic Charlotte, South Atlantic West, South Atlantic E, and S McNunn. The change originates from the CUST_REGION field in Bell's customer_data.csv, which feeds our daily ETL via Import.usp_ImportForecastCustomers.

This document provides a step-by-step plan to test the split in QADB_FORECAST before it goes live in production.

Critical Risk Existing forecasts will silently lose all South Atlantic data from the export pipeline after the split. The export view's INNER JOIN on RegionID returns 0 rows when locations move to new IDs. There is no error or warning — the data simply disappears from exports.

Silent Data Loss After the region split, existing forecasts lose all South Atlantic export data. Plan.WeeklyDistribution stores RegionID = 62 on every row. After the split, 0 locations have RegionID = 62. The INNER JOIN in the export view returns 0 rows. No error. No warning. Data just disappears from exports.

Can Old Forecasts Be Left Unchanged?

No. Not if they need to export correctly.
  • If we do nothing: Old forecasts silently lose SA data from exports. The numbers are still in the database, but the export pipeline can't find them.
  • If we only fix new forecasts: All active Cal_2026 forecasts still can't export SA data. You'd have to tell users "don't export any forecast created before the split" — that's not viable.
  • Exception: Truly archived forecasts that will never be exported again can be left as-is. The data persists; it just can't flow through the export pipeline.

Approaches Discussed (March 4 Sync w/ Eric Whitlow)

Four high-level approaches were discussed in the March 4 sync. These are the strategic options — the table-by-table migration details below apply to whichever in-place approach is chosen.

Approach 1: Separate Database ERIC'S PICK

"Probably the cleanest" — Eric Whitlow Copy everything to a new database. Delete old forecasts in the new DB. Set up new regions/locations as desired. Users pick which database to open — old DB for historical reference, new DB for going forward.
ProsCons
  • Old forecasts remain completely untouched in original DB
  • No migration script needed — new DB starts clean
  • No risk of corrupting existing data
  • Clean break: new regions, new forecasts, no legacy baggage
  • Two databases to maintain going forward
  • Users need to know which DB to open
  • New DB has no retail history for new sub-regions (same cold-start problem)
  • Any active Cal_2026 forecast work is stranded in old DB unless re-entered
  • Infrastructure cost of a second database

Approach 2: Rename + Create

Rename the existing "SOUTH ATLANTIC" region to one of the 4 sub-regions (e.g., "SA CLT"), then create the other 3 as new. Old forecast data partially works under the renamed region but only covers that sub-region's stores.

ProsCons
  • Partial continuity — one sub-region inherits all old data
  • Simpler than full proportional split
  • Only one sub-region gets the old data; other 3 start empty
  • The "lucky" sub-region gets inflated numbers (it inherited the full SA totals but only has ~30% of the stores)
  • Confusing for users — why does SA CLT have huge numbers and SA WEST has zero?

Approach 3: In-Place Migration (Proportional Split)

Keep the same database. Split existing SA forecast data proportionally across the 4 new sub-regions. See the table-by-table details below for how this works.

ProsCons
  • Preserves existing forecast work as a starting point
  • Single database — no user confusion
  • Users can review and adjust pre-populated numbers
  • Complex migration script (5 tables, proportional logic)
  • Proportional split is an approximation
  • Risk of bugs in migration corrupting data
  • Still has the retail history cold-start problem

Approach 4: New Attribute / Column

Add a sub-region dimension to the forecast data without changing the existing region structure. Deemed the trickiest and least clear on implementation.

ProsCons
  • Old data remains valid as-is
  • Backward compatible
  • Unclear how it would work end-to-end
  • Requires changes to views, procs, and possibly the UI
  • Most development effort
  • Eric flagged this as the trickiest

Migration: Table-by-Table (if doing in-place migration)

1. Plan.WeeklyDistribution

The problem: 14,736 rows (QADB) have RegionID = 62. After split, this ID has 0 locations. Export returns nothing.

The challenge: WeeklyDistribution is at the region level, not the location level. One row says "RegionID=62, ItemID=X, WeeklyQty=500." How do you split 500 across 4 sub-regions?

Option A: Proportional Split by Location Count

Divide existing quantities proportionally based on how many stores are in each sub-region.

Sub-RegionLocations% of 133Example: 500 units
SA CLT (NC)4030.1%150
SA WEST (TN+KY+IN)3828.6%143
SA EAST (SC+GA)2317.3%86
SA NONCORE (VA+WV)3224.1%121
-- Migration script: Split WeeklyDistribution rows proportionally
-- Run AFTER new regions exist and locations are reassigned

-- Calculate location-based proportions
WITH SubRegionCounts AS (
    SELECT
        l.RegionID AS NewRegionID,
        COUNT(*) AS LocCount,
        CAST(COUNT(*) AS DECIMAL(10,6)) /
            SUM(COUNT(*)) OVER() AS Proportion
    FROM Customer.Locations l
    WHERE l.RegionID IN (<SA_CLT_ID>, <SA_WEST_ID>, <SA_EAST_ID>, <SA_NONCORE_ID>)
    GROUP BY l.RegionID
)
-- Insert new split rows
INSERT INTO [Plan].WeeklyDistribution
    (ForecastID, RegionID, ItemID, WeekID, WeeklyQty, /* other columns */)
SELECT
    wd.ForecastID,
    src.NewRegionID,
    wd.ItemID,
    wd.WeekID,
    ROUND(wd.WeeklyQty * src.Proportion, 2) AS WeeklyQty
    /* other columns */
FROM [Plan].WeeklyDistribution wd
CROSS JOIN SubRegionCounts src
WHERE wd.RegionID = 62;

-- Then delete the old RegionID=62 rows
DELETE FROM [Plan].WeeklyDistribution WHERE RegionID = 62;
Trade-off Proportional split is an approximation. Different sub-regions may have different demand patterns (that's the whole reason for splitting). But it preserves existing forecast work and gives users a starting point to adjust.

Option B: Regenerate From Scratch

Users re-enter forecast numbers for all 4 sub-regions. Clean but loses all entered data for SA.

Option C: Hybrid (Recommended)

Split proportionally as a starting point, flag the forecasts as "migrated — review needed," and let users adjust the pre-populated numbers.

2. Plan.RegionForecasts

The problem: 283 rows reference RegionID = 62. If GenerateForecastRegions runs, it DELETEs these and creates empty rows for sub-regions. All user-entered forecast values are lost.

Schema reminder: RegionForecasts has 29 columns including Override, ForecastQty, and many week-level fields. These contain the actual numbers users have entered.

-- Migration: Capture SA values, then distribute to sub-regions

-- Step 1: Backup existing SA forecast values
SELECT * INTO #RF_SA_Backup
FROM [Plan].RegionForecasts
WHERE RegionID = 62;

-- Step 2: Run GenerateForecastRegions (creates empty sub-region rows)
-- Do this for EACH affected forecast
EXEC [Plan].GenerateForecastRegions @ForecastID = <ForecastID>;

-- Step 3: Proportionally fill the new sub-region rows from backup
-- (Same proportion logic as WeeklyDistribution)
UPDATE rf
SET
    rf.Override = ROUND(bk.Override * prop.Proportion, 2),
    rf.ForecastQty = ROUND(bk.ForecastQty * prop.Proportion, 2)
    /* ... repeat for each quantity column */
FROM [Plan].RegionForecasts rf
JOIN #RF_SA_Backup bk
    ON bk.ForecastID = rf.ForecastID AND bk.ItemID = rf.ItemID
JOIN SubRegionProportions prop
    ON prop.NewRegionID = rf.RegionID
WHERE rf.RegionID IN (<new sub-region IDs>);

3. Data.WeeklyRetailHistory

The problem: 3,555 rows of historical retail data (2019–2023) are tagged to RegionID = 62. New sub-regions have zero history. The ForecastWeeklyDistribution proc uses this history to calculate weekly patterns.

Option A: Copy SA History to All Sub-Regions (Recommended for now)

Each sub-region gets the same weekly pattern as the old SA aggregate. Inaccurate (the whole point of splitting is they're different), but functional.

-- Copy SA retail history to each new sub-region
INSERT INTO Data.WeeklyRetailHistory
    (YearID, WeekID, RegionID, ProductGroupID, WeeklyRetail, WeeklyRetailShipped)
SELECT
    wrh.YearID,
    wrh.WeekID,
    new_r.ID AS RegionID,
    wrh.ProductGroupID,
    ROUND(wrh.WeeklyRetail * prop.Proportion, 2),
    ROUND(wrh.WeeklyRetailShipped * prop.Proportion, 2)
FROM Data.WeeklyRetailHistory wrh
CROSS JOIN (
    SELECT l.RegionID,
           CAST(COUNT(*) AS DECIMAL(10,6)) /
           SUM(COUNT(*)) OVER() AS Proportion
    FROM Customer.Locations l
    WHERE l.RegionID IN (<new sub-region IDs>)
    GROUP BY l.RegionID
) prop
JOIN Customer.Regions new_r ON new_r.ID = prop.RegionID
WHERE wrh.RegionID = 62;
Long-term Once the sub-regions are live, real retail data will accumulate for each one. After a full year, the copied approximations can be replaced with actual data. Consider flagging these rows as "estimated" if the schema allows it.

Option B: Proportional Split by Location Count

Same as above but scales the retail values by sub-region store count proportions.

Option C: Reaggregate from Store-Level POS Data

If store-level retail data exists upstream, reaggregate by new sub-regions. Most accurate but requires access to source data we may not have.
